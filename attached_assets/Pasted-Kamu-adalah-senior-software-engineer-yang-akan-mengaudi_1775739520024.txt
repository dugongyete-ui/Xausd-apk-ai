Kamu adalah senior software engineer yang akan mengaudit dan memperbaiki proyek LIBARTIN — sebuah aplikasi sinyal trading XAUUSD berbasis React Native (Expo) + Node.js Express. Berikut audit lengkap dan daftar perbaikan yang perlu dilakukan:

═══════════════════════════════════════════════════
KONTEKS PROYEK
═══════════════════════════════════════════════════
- Stack: React Native (Expo) + Expo Router, Node.js Express backend
- Fungsi utama: menghasilkan sinyal entry trading XAUUSD dari data real-time Deriv API
- Strategi: Fibonacci retracement (50%–88.6%) dengan konfirmasi Pin Bar atau Engulfing di M5, berdasarkan EMA50 M15 sebagai filter trend utama
- File kunci:
  • server/derivService.ts    — engine sinyal (1351 baris)
  • server/aiService.ts       — integrasi AI Pollinations (623 baris)
  • server/routes.ts          — REST API endpoints (198 baris)
  • server/signalStore.ts     — penyimpanan sinyal ke file JSON lokal (44 baris)
  • contexts/TradingContext.tsx — state management + polling frontend (1526 baris)
  • app/(tabs)/index.tsx      — dashboard utama (996 baris)
  • app/(tabs)/signals.tsx    — histori sinyal (595 baris)
  • app/(tabs)/ai.tsx         — AI chat (714 baris)
  • components/FibChart.tsx   — visualisasi chart Fibonacci (883 baris)
  • scripts/backtest.ts       — backtesting CLI (1200 baris, terpisah dari app)
  • shared/marketRegime.ts    — deteksi ADX regime (trending/ranging/unknown)

═══════════════════════════════════════════════════
MASALAH 1 — LOGIKA SINYAL (server/derivService.ts)
═══════════════════════════════════════════════════

1a. M5_COUNT hanya 100 candle (~8 jam)
    - Terlalu sedikit untuk konteks yang baik, terutama saat market buka setelah weekend
    - Naikkan ke minimal 200 candle M5

1b. isCleanImpulse: batas span 3–25 bar M15 terlalu sempit
    - Impulse yang valid bisa berlangsung lebih dari 25 bar di kondisi trending kuat
    - Perlonggar batas atas ke 40 bar M15

1c. isCleanImpulse: minimum range hardcoded 5 poin
    - Untuk XAUUSD di harga ~4700 ini terlalu kecil (0.1% dari harga)
    - Ganti dengan ATR-relative: minimum range = 0.3 × M15 ATR

1d. Tidak ada batas waktu (expiry) sinyal di live app
    - Di backtest, sinyal expire setelah 60 bar M5 (~5 jam)
    - Di live app tidak ada mekanisme yang sama — sinyal "active" bisa bertahan selamanya
    - Tambahkan: jika sinyal aktif sudah lebih dari 5 jam sejak epoch dan tidak resolve, ubah status ke "expired" secara otomatis

1e. Tidak ada proteksi consecutive loss
    - Tambahkan MAX_DAILY_LOSS = 3 losses berturut-turut → hentikan sinyal hari itu (mode cooldown 4 jam)

1f. defaultBalance = 10000 hardcoded untuk lot size
    - Seharusnya bisa dikonfigurasi per user atau dari environment variable

MASALAH 2 — PERSISTENSI DATA (server/signalStore.ts)
═══════════════════════════════════════════════════

2a. Menggunakan file JSON lokal sebagai database
    - Data hilang saat server restart di environment ephemeral (cloud deploy)
    - Migrasi ke database: gunakan SQLite (via better-sqlite3) untuk deployment sederhana,
      atau PostgreSQL jika ingin skala lebih besar
    - Schema minimal: tabel signals dengan kolom: id, pair, trend, entryPrice, stopLoss,
      takeProfit, takeProfit2, rr1, rr2, status, outcome, signalEpoch, resolvedEpoch,
      confirmationType, sessionTag, confluence, marketRegime, createdAt

2b. signalStore.ts hanya 44 baris, tidak ada error handling
    - Tambahkan try/catch di semua operasi baca/tulis
    - Tambahkan validasi schema sebelum menyimpan

═══════════════════════════════════════════════════
MASALAH 3 — AI SERVICE (server/aiService.ts)
═══════════════════════════════════════════════════

3a. Menggunakan Pollinations AI (API publik gratis)
    - Tidak reliable untuk production — sering timeout, tidak ada SLA
    - Tambahkan fallback: jika Pollinations gagal 2x berturut-turut, switch ke mode
      "template response" berbasis data pasar tanpa AI
    - Atau: dukung konfigurasi provider via env var (POLLINATIONS / OPENAI / GROQ)

3b. Tidak ada rate limiting di AI chat
    - User bisa spam request → server overload
    - Tambahkan limit: maks 5 pesan per menit per session (in-memory rate limiter)

3c. Konteks AI tidak menyertakan outcome sinyal sebelumnya
    - AI tidak tahu apakah 3 sinyal terakhir win atau loss
    - Tambahkan ke context string: "5 sinyal terakhir: WR X%, losses terakhir: N berturut-turut"

3d. Timeout AI hardcoded 35 detik
    - Terlalu lama untuk UX mobile
    - Turunkan ke 20 detik dengan retry otomatis 1x

MASALAH 4 — FRONTEND ARCHITECTURE
═══════════════════════════════════════════════════

4a. TradingContext.tsx terlalu besar (1526 baris)
    - Pecah menjadi beberapa custom hooks:
      • useMarketState()      — data market, EMA, Fibonacci, regime
      • useSignalManager()    — sinyal aktif, histori, outcome tracking
      • useWebSocketPrice()   — koneksi WS real-time ke backend atau Deriv langsung
      • useSessionStatus()    — london/NY session, market hours

4b. Polling interval 15 detik untuk sinyal
    - Bisa melewatkan event penting (sinyal muncul + langsung kena SL dalam < 15 detik)
    - Gunakan Server-Sent Events (SSE) dari backend untuk push update sinyal secara real-time
    - Backend sudah ada /api/ai/stream — pola yang sama bisa dipakai untuk sinyal

4c. ai.tsx polling 2 detik untuk AI response
    - Backend sudah ada streaming endpoint — gunakan langsung fetch streaming
      bukan polling interval terpisah

4d. FibChart.tsx terlalu besar (883 baris) dan murni SVG
    - Potensi jank di device low-end karena re-render setiap tick harga
    - Memoize komponen dengan React.memo + useMemo untuk level Fibonacci
    - Pisahkan: FibChart.tsx → FibZoneRenderer.tsx + PriceTracker.tsx + SwingLabels.tsx

4e. Tidak ada error boundary di React Native app
    - Tambahkan <ErrorBoundary> di level root _layout.tsx untuk catch crash gracefully

4f. Tidak ada loading skeleton
    - Saat pertama load, semua data null → UI terlihat kosong/broken
    - Tambahkan skeleton placeholder di dashboard, signals list, dan chart

MASALAH 5 — BACKEND ARCHITECTURE

    - Pindahkan ke config file: server/config.ts dengan komentar yang jelas

5c. Tidak ada health check endpoint
    - Tambahkan GET /api/health yang return:

    - Useful untuk monitoring deployment

5d. Deriv WebSocket tidak ada reconnect exponential backoff

    - Tambahkan exponential backoff: 1s, 2s, 4s, 8s, max 30s

═══════════════════════════════════════════════════
MASALAH 6 — STRATEGI & SIGNAL QUALITY (backtest-driven)
═══════════════════════════════════════════════════

6a. Backtest 30 hari hanya menghasilkan 11 sinyal (~0.37/hari)
    - Frekuensi terlalu rendah untuk validasi statistik yang baik (butuh >30 sinyal)
    - Pertimbangkan tambah timeframe M1 sebagai trigger konfirmasi akhir setelah
      M5 sudah memenuhi syarat zona Fibonacci

6b. Engulfing pattern masih menghasilkan lebih banyak loss daripada rejection
    - Di backtest terbaru: Engulfing 50% WR vs Pin Bar 67% WR
    - Pertimbangkan: engulfing hanya valid jika juga terjadi pada level round number
      (confluence wajib untuk engulfing, opsional untuk rejection)

6c. Session filter belum optimal
    - Active session (London+NY) kadang WR lebih rendah dari low_confidence
    - Kemungkinan karena volatilitas tinggi di London open justru sering spike melewati SL
    - Pertimbangkan exclude periode London open pertama 30 menit (07:00–07:30 UTC)
      sebagai "spike zone" dan New York open pertama 30 menit (13:00–13:30 UTC)

6d. Tidak ada multi-timeframe confluence yang ketat
    - Saat ini: M15 EMA50 + M5 EMA20/50 + M5 pattern
    - Tambahkan: M30 atau H1 trend sebagai konfirmasi tambahan
      (harga di atas EMA20 H1 untuk bullish, di bawah untuk bearish)

6e. TP2 formula (Fib 127.2% + 0.5×ATR) kadang menghasilkan target terlalu jauh
    - Jika range swing besar (>100 poin), TP2 bisa 200+ poin dari entry → tidak realistis untuk M5
    - Tambahkan cap: TP2 maksimal 3× ATR M15 dari entry price

MASALAH 7 — KEAMANAN & DEPLOYMENT
═══════════════════════════════════════════════════

7a. Tidak ada autentikasi sama sekali
    - Semua endpoint /api/* terbuka tanpa token/key
    - Minimal: tambahkan API_KEY sederhana di header X-Api-Key untuk semua endpoint
    - Lebih baik: JWT atau Expo Auth untuk multi-user

7b. Tidak ada CORS policy yang ketat
    - Saat ini menerima semua origin
    - Di production, batasi ke domain Expo app saja

7c. Push notification token tidak ada validasi
    - Expo push token langsung digunakan tanpa validasi format
    - Tambahkan validasi: token harus match format ExponentPushToken[...]
